# Wiki Research Scout

## 一句话定位

给代理自己用的 wiki 研究选题前置探索 skill。
负责在 `wiki-kb-builder` 之前做一轮"发现与侦察"：
- 从现有 wiki 中找出值得深入的知识点
- 去外部站点搜集初步资料
- 形成结构化候选清单，等用户确认后再进后续流程

**它不建页、不修链、不改库、不下载。** 建页交给 `wiki-kb-builder`，下载外部资料要等用户确认后路由给对应 skill（`paper-replication` / `neodata-financial-search` / `wechat-article-crawler`）。

---

## 工作流程（四段式）

### Phase 1：扫描 wiki-kb，识别候选扩展点

**输入**：`<YOUR_WIKI_KB_PATH>`（wiki 根目录）

从 6 个维度扫描，具体定义参见 `references/topic-patterns.md`：

1. **高频被引用但内容薄弱** — 页面存在但正文简短，支撑不起引用量
2. **高频断链概念** — `[[概念]]` 多处出现但无独立页面
3. **单页过重** — 一页包含多个可独立展开的子主题
4. **主题簇缺口** — 已有相关页簇，但缺关键枢纽/总览/对比页
5. **跨主题交叉机会** — 两个主题簇有共现但无桥接页
6. **陈旧页补新进展** — 页面久未更新且主题仍在演进

**输出**：候选主题列表（每个含触发原因、关联页面、推荐搜索词、推荐动作类型）

---

### Phase 2：对外部资料进行侦察

对每个候选主题按分类分配搜索源，具体源选择规则见 `references/site-selection-policy.md`：

| 主题类型 | 优先站点 |
|---|---|
| 量化因子/策略/论文 | arXiv, SSRN, Google Scholar, PapersWithBacktest |
| 价格行为/技术分析/交易心理 | 公众号, 博客, Medium, Substack, 专题网站 |
| 宏观/行业/资产配置 | 券商研报, 财经媒体, 研究机构博客 |
| 泛知识/经验类 | 知乎, 雪球, 专栏博客 |

侦察动作：
1. 生成适配的搜索 query（中英文各一版）
2. 用 web_search 检索候选来源
3. 滤除低质量、重复来源
4. 按来源可信度分级（A/B/C）
5. 每主题提炼 3~10 条值得看的外部来源（初侦察阶段：2 个站点通常可返回 5~15 条结果，经质量过滤后保留 3~5 条；深搜阶段：3~5 个站点可覆盖更多 A 级来源）

**输入**：Phase 1 候选列表
**输出**：加注外部来源的候选卡片

---

### Phase 3：形成候选清单，等用户确认

使用 `references/output-template.md` 定义的结构化格式。

最终产出一个研究报告（Markdown 格式），包含：
- 总览（扫描范围、候选数量、优先级分布）
- 每个候选主题的完整卡片
- 下一步建议（推荐动作 + 推荐衔接的 skill）

---

### Phase 4：下载确认与路由分发

用户查看 Phase 3 候选清单后，**必须**主动询问：

**询问话术模板**（精简版）：
```
这些候选中，需要我下载外部资料吗？
候选涉及的资料类型：
  类型A · 论文/学术资料（X 个候选涉及） → 走 paper-replication
  类型B · 券商研报/财经资讯/量化报告（X 个候选涉及） → 走 neodata-financial-search
  类型C · 公众号文章（X 个候选涉及）    → 走 wechat-article-crawler

请选择：
  A) 下载全部论文资料（用 paper-replication 提取摘要/代码）
  B) 下载全部研报资料（用 neodata 搜索研报）
  C) 下载全部公众号文章（用 wechat-article-crawler 抓取）
  D) 自定义选择（指定哪些候选/哪些类型）
  E) 暂不下载，直接入库建页（跳至 wiki-kb-builder）
  F) 以上都不做，结束本次侦察
```

**路由规则**（根据用户选择的资料类型启动对应 skill）：

| 用户选择的资料类型 | 启动的 skill | 传递内容 |
|---|---|---|
| 学术论文 / journal / arXiv | `paper-replication` | 候选标题 + URL 列表 + 研究方向 |
| 券商研报 / 财经资讯 / 量化研究报告 | `neodata-financial-search` | 搜索关键词 + 候选主题描述 |
| 公众号文章 | `wechat-article-crawler` | 候选文章标题 + URL（如有）|
| 混合类型（多种同时选） | 并行启动对应 skill | 分别路由，各自传递 |

**执行规则**：
1. **必须先询问，不能默认下载**。尊重 suggest-only 默认模式。
2. 来源分类依据 Phase 2 中标注的可信度等级和来源站点类型来判断：
   - arXiv/SSRN/Google Scholar/PapersWithBacktest → 论文 → 路由给 `paper-replication`
   - 券商研报 / 财经资讯 / 量化策略分析文章 → 路由给 `neodata-financial-search`
     - 实测 `neodata` 的 `docData` 覆盖两类：`type="公司研报"`（券商正式研报）和 `type="自建资讯搜索"`（财经资讯/金工量化报告），两者均可作为研究素材
     - 另可通过 `apiData` 获取行情、财报、板块、基金、宏观等结构化数据
   - 公众号/专题网站 → 公众号文章 → 路由给 `wechat-article-crawler`
   - 知乎/雪球/论坛/博客 → 标注"无专用 skill"，建议用户手动处理
3. 用户选 E（仅入库建页）或 F（结束）时，不启动任何下载 skill。
4. 路由时只传递 URL 列表 + 方向描述，**不代执行**下载/搜索动作——下载由目标 skill 的完整流程负责。

---

## 输出格式

严格遵循 `references/output-template.md` 中定义的结构。

每个候选卡片必须包含：
- 标题
- 类型（新建 / 扩写 / 拆页 / 桥接）
- 优先级（高 / 中 / 低）
- wiki 内部证据（引用频次、关联页列表、触发维度）
- 外部来源候选（标题、URL、可信度等级、初步判断）
- 建议下一步

---

## 来源可信度规则

定义参见 `references/site-selection-policy.md`，核心约束：

- **A级**（高可信）：论文、券商研报、研究机构正式发布 → 可作主证据
- **B级**（中可信）：知名博客、Substack、专业公众号 → 可作解释/案例
- **C级**（弱可信）：知乎、雪球、论坛 → 仅作线索，不独立支撑结论

输出时必须标注每条来源的评级。

## 默认模式

**suggest-only**。除非用户明确要求更激进执行，否则只出建议不改库。

---

## 执行约束协议

### Phase 1 约束
- **最大候选数上限**：10 个。超出时只展示高优先级 + 标注"另有 X 个中/低优先级候选待展开"。
- **最低候选门槛**：Phase 1 扫描产出 < 2 个候选时，直接标记"暂无值得深挖的方向"，不进入 Phase 2。

### Phase 2 约束（初侦察）
- **每候选最大侦察站点数**：2 个。优先使用 site-selection-policy 中优先级最高的站点，第二个站点用于交叉验证。
- **搜索失败降级策略**：
  - 首个站点无结果 → 换第二站点
  - 两站点均无结果 → 候选卡片标注"暂未找到高质量外部来源"，仍展示 wiki 内部证据
  - 不强行关联不匹配的来源
- **循环检查**：连续 2 个候选均无外部来源时，主动提示用户"本批次源质量较低，是否扩大搜索范围或先跳过？"

### Phase 2b 约束（用户确认后的深度搜索）
- **触发条件**：用户从初侦察候选中确认具体方向后
- **每候选搜索源数**：3~5 个。优先覆盖 A/B 级来源，允许跨主题类型搜索
- **搜索预算**：深度搜索不占用初侦察的 20 次预算，单独计数，上限 15 次

### 输出约束
- 单次报告 ≤10 个候选卡片（超出部分标注编号 + 优先级 + 简略理由，不展开）
- 用户确认高优先级方向后，再为确认的方向做二次深度搜索（Phase 2b 执行，每候选搜 3~5 个源）

### 会话级约束
- **初始运行预算**：20 次（覆盖 Phase 1 扫描 + Phase 2 初侦察的 web_search 调用）
- **深度搜索预算**：15 次（仅用于用户确认后的 Phase 2b，独立计数）
- 初始运行达 20 次上限后，报告当前进度的候选清单 + 标注"已达初始搜索预算上限，剩余候选未侦察"
- 深度搜索达 15 次上限后，标注"深度搜索预算已用尽"
- 用户要求续搜时重置对应阶段的预算

## 交互准则

面向用户时：
- 先说发现「X 个候选主题，高优先级 Y 个」，不直接丢细节
- 用户要看细节再展开
- 不要报 PowerShell/CLI 用法
- 把候选主题列成选择题让人圈定范围
- Phase 4 必须主动询问是否需要下载资料，询问模板见 Phase 4，但精简改写成选择题式

---

## 安全 / 质量规则

1. **不会就是不会** — 对不熟悉的研究领域，标注这是探索性搜索
2. **区分事实与推测** — 外部来源的结论是"据某文称"，不是引用就自动等于事实
3. **禁幻觉** — 找不到来源就直接说没找到，不编造 URL
4. **禁越权** — 不改 wiki-kb，不自动建页，不自动下载。下载必须经用户确认再路由。
5. **来源可见** — 每条外部参考必须给出来源标题和 URL
6. **批处理需用户确认** — 候选列表超 10 项时，先问用户要不要只看高优先级
